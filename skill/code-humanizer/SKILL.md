# Code Humanizer

Companion to **kyoko-humanize**. Same philosophy — strip AI performance, restore
human intent — but targets code (and the prose around it: commits, PR descriptions)
instead of essays.

## Relation to the other skills

- **Prose** → use [`kyoko-humanize`](../kyoko-humanize/SKILL.md)
- **Code** → use this skill
- **Both** → run the auditor/Kyoko on comments and docs first, then this on code structure
- **Scoring philosophy** → shared with [`ai-writing-auditor`](../ai-writing-auditor/SKILL.md): the output is a pass-based **verdict** (PASS / REVISE / FAIL), framed as **AI-tell risk under a rubric, not a judgment on who wrote it.** Clean code from a careful human and clean code from a model can look identical; absence of tells is not proof of authorship, and presence is not proof of a problem.

## How AI code reads (2025–2026)

The recognizable tells beyond raw vocabulary:

- **Explicit attribution markers** — `Co-Authored-By` trailers, "🤖 Generated with…" footers, model-name lines. Highest-confidence and trivially strippable, so their *absence* means nothing.
- **Tutorial verbosity** — comments restating the next line, all-sections-present docstrings on trivial functions, placeholder scaffolding left in (`# TODO: implement`, `# ... rest of implementation`).
- **Defensive overkill** — try/catch around code that can't throw, null checks on internally-controlled values; this *hides* real bugs.
- **Over-abstraction** — single-implementation interfaces, `*Impl` suffixes, factories/wrappers with one caller; "enterprise" patterns where a function would do.
- **Redundancy** — near-duplicate logic blocks (Type-4 clones) instead of reuse.
- **Too-smooth uniformity** — perfectly even comment cadence, uniform abstraction level, no idiosyncrasy. The meta-tell.

## Pipeline

Run stages in order. Each is independent — skip if not applicable.

---

### Stage 0 — `scrubAttributionAndArtifacts`

Catch the cheapest, highest-confidence tells first.

**Remove / flag:**
- `Co-Authored-By: Claude <noreply@anthropic.com>` (and Cursor/Copilot equivalents) trailers in commit messages
- Model-name lines and `🤖 Generated with [Claude Code]` / "Generated by" footers in commit bodies and PR descriptions
- Session/trace links pasted into commit or PR text

**Note:** these are config-controlled, so their absence proves nothing. Only remove them when the user actually wants the attribution gone — do not strip a `Co-Authored-By` trailer the user intends to keep. When in doubt, ask.

---

### Stage 1 — `stripObviousComments`

Remove comments that restate the adjacent code, and leftover scaffolding.

**Remove:**
- Line comments describing what the next line does: `// increment i`, `// return result`
- Empty TODOs with no body: `// TODO: handle error`
- Placeholder/debug residue: `# TODO: implement`, `# ... rest of implementation`, `# Your code here`, stray `print()` / `console.log` debug in production paths
- Docblocks on trivial getters/setters with no params, no throws, obvious return
- Section dividers with no semantic meaning: `// ---- helpers ----`

**Keep:**
- Non-obvious intent: *why* a value is hardcoded, *why* an order matters
- Workarounds with ticket refs
- Public API docblocks (parameter contracts, throws, examples)

---

### Stage 2 — `collapseNeedlessHelpers`

Inline single-use extracted functions that add no abstraction value.

**Collapse when:**
- Function called exactly once
- Body ≤ 5 lines
- Not recursive, not async boundary, not a named test fixture
- Name doesn't encode a domain concept (just describes the lines it wraps)

**Keep when:**
- Called 2+ times
- Name is a domain term (`validateLead`, `buildQuery`)
- Extracted for testability (unit test references it directly)

---

### Stage 3 — `collapseFalseAbstractions`

Distinct from Stage 2: not "small helper," but *ceremony* — structure that exists only because the pattern was recognizable in training data.

**Collapse / flatten:**
- Single-implementation interfaces with one concrete class (`Foo` + `FooImpl`, one implementor) — inline to the concrete type
- Factories, builders, wrappers, or adapters with exactly one caller and no polymorphism
- Manager/Service layers that only forward calls to one other object
- Config objects with one field passed to one function

**Keep:**
- Interfaces with 2+ implementations, or a real seam for testing/mocking
- Patterns the surrounding codebase already uses consistently (match the house style)

**Rule:** YAGNI. If the indirection has exactly one path through it today, it's scaffolding.

---

### Stage 4 — `deflateNames` + thin redundant annotations

Shorten names inflated with redundant type/role tokens; drop annotations that inference and surrounding style already cover.

| Pattern | Example | Fix |
|---------|---------|-----|
| Type in name | `userData` | `user` |
| Redundant noun | `getUserById` | `findUser` |
| Event suffix | `handleClickEvent` | `onClick` |
| Manager/Helper suffix | `UserDataManager` | `Users` or `UserStore` |
| Boolean prefix stutter | `isUserLoggedIn` | `isLoggedIn` (in User context) |
| Over-descriptive identifier | `calculate_total_price_including_tax_amount` | `totalWithTax` |
| Redundant local annotation | `const n: number = items.length` | `const n = items.length` |

**Rule:** names should be as short as the call site makes unambiguous; annotate only where inference doesn't reach and the surrounding code annotates too.

---

### Stage 5 — `pruneRedundantGuards`

Remove guards that aren't reachable given the actual data flow — and redundant defensive density even where individual guards are reachable.

**Remove:**
- Null checks on values the type system guarantees non-null
- `try/catch` wrapping code that cannot throw in practice (and stacked try/catch where one boundary suffices)
- Unreachable `else` branches that exist only for visual symmetry
- `|| null` / `|| undefined` on already-nullable values
- `=== undefined || === null` → `== null` where appropriate

**Keep:**
- Guards at public API boundaries (external input)
- Checks where the type system doesn't reach (untyped legacy code)
- Error handling that logs or transforms (not silent swallowing)

**Watch for:** swallowed errors — AI defensive code often `catch`es and returns a default, hiding real failures. Surfacing the error is more human than hiding it.

---

### Stage 6 — `deflateDocstrings`

Break templated "perfect completeness" — every standard section present on every function, however trivial.

**Deflate:**
- Drop `Args/Returns/Raises/Examples` sections on functions where the signature already says it (trivial getters, one-line helpers)
- Remove `Raises:` / `Examples:` sections that add nothing
- Break uniform templated phrasing — vary or cut, don't restate the function name as a sentence

**Keep:**
- Real public-API contracts: non-obvious parameter semantics, actual thrown errors, units, edge cases
- Examples that show genuinely non-obvious usage

---

### Stage 7 — `deduplicateClones`

Flag near-duplicate (Type-4) logic blocks — AI tends to regenerate similar logic rather than reuse.

**Look for:** two+ blocks doing the same thing with renamed variables or trivial structural differences. Extract a shared function **only when** it's a real domain concept (not just textual overlap) and the extraction reads cleaner than the duplication. Otherwise leave it — premature de-duplication is its own AI tell.

---

### Stage 8 — `matchSurroundingStyle`

Infer the style profile from context files and apply it.

**Profile points:** quote style, semicolons, trailing commas, brace style (`fn()` vs `fn ()`, `{x}` vs `{ x }`), arrow style (`x => x` vs `(x) => x` vs `function(x)`).

**Source:** 2–3 files the human owns in the same module. Never apply the project's prettier config blindly — match what they *actually write*, which may diverge from config.

---

### Stage 9 — `breakUniformity` *(hardest)*

Inject natural imperfection. Humans optimize for readability of *their mental model*, not spec completeness — and they're not uniform. This stage targets the too-smooth signature directly.

**Transforms:**
- Drop redundant parens where precedence is obvious: `(x * 2)` → `x * 2`
- Collapse pointless ternary: `x ? true : false` → `!!x` or just `x`
- Drop `return undefined` / `return void` at the end of void functions
- Remove braces on single-statement `if` when team style allows
- `arr.length` for truthiness instead of `arr.length > 0`
- Prefer `+=` / `-=` over `x = x + …`
- Vary comment cadence and abstraction level — humans don't comment every block evenly
- Consolidate chained `.filter().map()` only when the combined form reads cleaner

**Do NOT:**
- Introduce bugs for the sake of looking human
- Change semantics — only equivalent rewrites
- Apply all transforms — pick the ones matching the surrounding file's slop profile

---

### Stage 10 — `humanizeCommitAndPrProse`

When the diff ships with a commit message or PR description, those are prose — apply the auditor's lens.

**Deflate:**
- Cap emoji density (AI agents bullet with emoji "to enhance readability")
- Reduce markdown header / bullet count; a small diff doesn't need a multi-section PR template
- Shorten over-explanation — don't narrate a 3-line change in 4 paragraphs
- Drop conventional-commit theater on trivial diffs

Hand longer prose bodies to [`ai-writing-auditor`](../ai-writing-auditor/SKILL.md) for a full pass.

---

## Verdict (pass-based; the number is optional)

After the pipeline, emit a **verdict**, not a number — **as AI-tell risk under this rubric, not a judgment on authorship.**

| Verdict | When | Meaning |
|---------|------|---------|
| **PASS** | No attribution leakage, no false abstractions or dead guards left, style matches the surrounding human code | Reads like this codebase. Hand back. |
| **REVISE** | Only minor tells remain (a stray tutorial comment, uneven comment cadence) | One more pass if it's worth it. |
| **FAIL** | Attribution markers the user wanted gone, surviving false abstractions / redundant guards, or style that clashes with the module | Re-run the relevant stages. |

State which stages fired and a one-line limitations note (clean code is not proof of human authorship; this measures tells, not origin). Add an optional 0–100 score only if asked (PASS ≈ 85–100, REVISE ≈ 60–84, FAIL ≈ 0–59).

## Adaptive Loop (multiple passes; findings-driven)

- `max_passes`: 3 — the loop iterates, converging on the verdict and remaining tells, not a score target.
- Stop: verdict is **PASS**, or passes hit the cap, or the remaining tells don't shrink two passes in a row.
- Rollback if Stage 9 changes semantics (revert that stage only).
- Structural-first: collapse false abstractions and prune guards (Stages 3, 5) before chasing Stage 9 polish — they decide the verdict.

## Limitations

Stages 7 and 9 (`deduplicateClones`, `breakUniformity`) need an author fingerprint. Without
git history or surrounding human-written code, use conservative defaults — apply only
universally-safe transforms, skip opinionated ones.

Full fingerprinting requires:
```
inferAuthorIntent(ast, gitBlame, surroundingCode) → styleFingerprint
```
A fine-tuned model problem: train on diffs between raw AI output and human-edited
versions of the same code. Not rule-based alone.
